# map this into a concrete RBAC design for your AML platform with example PostgreSQL tables and Keycloak realm role

For your AML platform, use **Keycloak realm roles for global platform identity**, **Keycloak client roles for app-specific capabilities**, and **PostgreSQL for tenant-scoped role assignments, case access, and row-level enforcement**. That gives you clean SSO and token issuance in Keycloak, while keeping the actual business authorization model close to your AML data and workflows in Postgres.[^1][^2][^3][^4]

## Role model

A practical AML workflow needs segregation among investigator, reviewer, approver, and platform admin roles, with action auditing and case-state controls. That lines up with standard AML case-management role matrices, where investigation, review, escalation, and approval are separated for trust and chain-of-custody reasons.[^5][^6]

Use this split:

- **Realm roles** for global identity status across the platform, such as `platform_admin`, `tenant_admin`, `compliance_officer`, `auditor`, `api_service`.[^1][^7]
- **Client roles** for app features inside the AML application client, such as `alerts.read`, `alerts.review`, `cases.create`, `cases.assign`, `cases.approve`, `graph.explore`, `rules.manage`, `reports.file_str`, `users.manage`.[^8][^3][^1]


## Keycloak design

Create one realm for the platform, for example `overwatch-aml`, and define realm roles only for roles that must be recognized across multiple clients such as frontend, API, ETL admin, and reporting services. Keycloak’s model distinguishes global realm roles from more granular client roles, and you can combine them in policies when needed.[^1][^8][^3]

Suggested realm roles:

- `platform_admin`
- `tenant_admin`
- `compliance_officer`
- `auditor`
- `investigator_lead`
- `service_etl`
- `service_rules_engine`[^7][^1]

For the AML app client, suggested client roles:

- `dashboard.view`
- `alerts.view`
- `alerts.triage`
- `alerts.escalate`
- `cases.view`
- `cases.create`
- `cases.assign`
- `cases.edit`
- `cases.review`
- `cases.approve`
- `cases.close`
- `evidence.upload`
- `graph.view`
- `graph.expand`
- `screening.view`
- `screening.review`
- `str.create`
- `str.approve`
- `users.view`
- `users.manage`
- `rbac.manage`
- `rules.view`
- `rules.manage`
- `etl.monitor`[^3][^1]

A clean pattern is to create Keycloak composite roles such as `aml_investigator`, `aml_reviewer`, and `aml_manager`, then bundle the client roles into each composite role. Keycloak supports combining roles this way so your applications consume simpler role sets while preserving fine-grained permissions underneath.[^9][^3]

## PostgreSQL schema

Use row-based multi-tenancy with `tenant_id` on all tenant-owned data and Row-Level Security for isolation. PostgreSQL multi-tenant guidance commonly uses shared tables with tenant identifiers and RLS to prevent leakage across tenants.[^2][^4][^10]

Example core identity and RBAC tables:

```sql
create extension if not exists pgcrypto;

create table tenants (
  tenant_id uuid primary key default gen_random_uuid(),
  tenant_code text not null unique,
  tenant_name text not null,
  status text not null default 'active',
  created_at timestamptz not null default now()
);

create table app_users (
  user_id uuid primary key default gen_random_uuid(),
  keycloak_user_id uuid not null unique,
  email text not null unique,
  full_name text,
  status text not null default 'active',
  created_at timestamptz not null default now()
);

create table tenant_memberships (
  tenant_id uuid not null references tenants(tenant_id) on delete cascade,
  user_id uuid not null references app_users(user_id) on delete cascade,
  membership_status text not null default 'active',
  joined_at timestamptz not null default now(),
  primary key (tenant_id, user_id)
);

create table roles (
  role_id uuid primary key default gen_random_uuid(),
  role_code text not null unique,
  role_name text not null,
  role_scope text not null check (role_scope in ('realm','tenant','case')),
  description text
);

create table permissions (
  permission_id uuid primary key default gen_random_uuid(),
  permission_code text not null unique,
  resource_type text not null,
  action text not null,
  description text
);

create table role_permissions (
  role_id uuid not null references roles(role_id) on delete cascade,
  permission_id uuid not null references permissions(permission_id) on delete cascade,
  primary key (role_id, permission_id)
);

create table user_tenant_roles (
  tenant_id uuid not null references tenants(tenant_id) on delete cascade,
  user_id uuid not null references app_users(user_id) on delete cascade,
  role_id uuid not null references roles(role_id) on delete cascade,
  granted_by uuid references app_users(user_id),
  granted_at timestamptz not null default now(),
  primary key (tenant_id, user_id, role_id)
);
```

This structure separates users, tenants, roles, and permissions cleanly, which is the standard foundation for tenant-aware RBAC in PostgreSQL.[^4][^2]

## AML case controls

AML platforms usually need both tenant-level and case-level access because some users can see all alerts in a tenant while others can only work assigned cases or review queues. Mature AML workflows also require investigator, reviewer, and approver segregation plus audit trail on every status change.[^5]

Add case-scoped tables like this:

```sql
create table cases (
  case_id uuid primary key default gen_random_uuid(),
  tenant_id uuid not null references tenants(tenant_id) on delete cascade,
  case_number text not null unique,
  status text not null check (status in ('open','under_review','approved','filed','closed')),
  severity text not null check (severity in ('low','medium','high','critical')),
  created_by uuid not null references app_users(user_id),
  assigned_to uuid references app_users(user_id),
  reviewer_id uuid references app_users(user_id),
  approver_id uuid references app_users(user_id),
  created_at timestamptz not null default now()
);

create table case_acl (
  case_id uuid not null references cases(case_id) on delete cascade,
  user_id uuid not null references app_users(user_id) on delete cascade,
  access_level text not null check (access_level in ('view','edit','review','approve')),
  granted_by uuid references app_users(user_id),
  granted_at timestamptz not null default now(),
  primary key (case_id, user_id, access_level)
);

create table audit_access_events (
  event_id uuid primary key default gen_random_uuid(),
  tenant_id uuid,
  user_id uuid,
  resource_type text not null,
  resource_id uuid,
  action text not null,
  decision text not null check (decision in ('allow','deny')),
  reason text,
  created_at timestamptz not null default now()
);
```

This lets you mix role-based rights with explicit case assignment when a sensitive investigation should only be visible to named users.[^2][^5]

## Example mappings

Map Keycloak composite roles to PostgreSQL application roles like this:


| Business role | Keycloak realm role | Keycloak client roles | PostgreSQL app role meaning |
| :-- | :-- | :-- | :-- |
| Platform admin | `platform_admin` [^7] | `users.manage`, `rbac.manage`, `rules.manage`, `etl.monitor` [^3] | Cross-tenant admin with full platform control. [^2] |
| Tenant compliance admin | `tenant_admin` [^7] | `dashboard.view`, `alerts.view`, `cases.view`, `users.manage`, `rbac.manage`, `reports.file_str` [^3] | Admin for one institution or tenant. [^2][^4] |
| AML investigator | none or `investigator_lead` for leads [^7] | `alerts.view`, `alerts.triage`, `cases.create`, `cases.edit`, `evidence.upload`, `graph.view`, `graph.expand` [^3] | Works alerts and investigations, no final approval. [^5] |
| AML reviewer | `compliance_officer` optional [^7] | `alerts.escalate`, `cases.review`, `screening.review`, `str.create` [^3] | Reviews analyst work and escalates. [^5] |
| AML approver | `compliance_officer` [^7] | `cases.approve`, `cases.close`, `str.approve` [^3] | Final decision maker for suspicious activity filing. [^5] |
| Auditor | `auditor` [^7] | `dashboard.view`, `cases.view`, `alerts.view` [^3] | Read-only access with audit visibility. [^5] |

## Authorization flow

At login, Keycloak issues the JWT containing realm roles and client roles. Your API then resolves the Keycloak user ID to `app_users.user_id`, loads tenant memberships and tenant role grants from PostgreSQL, and enforces both token roles and database state before granting access.[^1][^7][^2]

In practice:

1. Verify JWT from Keycloak.[^3]
2. Read realm roles and client roles from token claims.[^7][^1]
3. Resolve `tenant_id` from request context.
4. Check `tenant_memberships` and `user_tenant_roles` in PostgreSQL.
5. Apply case-level ACL checks for sensitive cases.
6. Log allow or deny into `audit_access_events`.[^5][^2]

## RLS pattern

Use PostgreSQL RLS on tenant-owned tables so app bugs do not accidentally leak cross-tenant data. Row-based multi-tenancy with `tenant_id` plus RLS is a standard way to protect shared-schema SaaS data.[^2][^4][^10]

Example:

```sql
alter table cases enable row level security;

create policy cases_tenant_isolation
on cases
using (
  tenant_id = current_setting('app.current_tenant')::uuid
);

create policy cases_assigned_or_privileged
on cases
using (
  tenant_id = current_setting('app.current_tenant')::uuid
  and (
    assigned_to = current_setting('app.current_user')::uuid
    or reviewer_id = current_setting('app.current_user')::uuid
    or approver_id = current_setting('app.current_user')::uuid
    or exists (
      select 1
      from user_tenant_roles utr
      join roles r on r.role_id = utr.role_id
      where utr.tenant_id = cases.tenant_id
        and utr.user_id = current_setting('app.current_user')::uuid
        and r.role_code in ('tenant_admin','aml_manager','auditor')
    )
  )
);
```

This pattern keeps tenant isolation in the database while still allowing privileged users broader access inside their own tenant.[^4][^2]

## Recommended final shape

Use Keycloak for:

- Authentication, MFA, SSO, IdP federation, token issuance.
- Global realm roles.
- AML app client roles and composite roles.[^1][^3]

Use PostgreSQL for:

- Tenant membership.
- Effective role grants by tenant.
- Case assignment and case ACL.
- Audit evidence for every authorization decision.
- RLS-based tenant isolation.[^5][^2][^4]

That combination is the right fit for your AML platform because AML operations need both classic RBAC and workflow-sensitive controls such as reviewer separation, approval gates, and case-by-case restrictions.[^6][^5]
