# turn this into a full SQL starter pack and Keycloak realm export structure

I created a full starter pack with PostgreSQL DDL, seed RBAC data, RLS policies, helper functions, bootstrap data, and a Keycloak realm import skeleton tailored for your AML platform. The package includes SQL files for tenants, users, roles, permissions, alerts, cases, case ACL, and audit logging, plus a realm JSON and a composite-role plan for Keycloak.[^1][^2][^3][^4]

## What’s included

The SQL pack is split into ordered files so you can apply schema, seed data, RLS, helper functions, and sample bootstrap data in sequence, which matches common PostgreSQL multi-tenant and RLS patterns using `current_setting('app.current_tenant')` for tenant context.  The Keycloak side includes an importable realm skeleton with realm roles, AML API client roles, token mapping for `tenant_id`, and a follow-up composite-role plan because Keycloak role composition is commonly managed with realm and client role layering.[^5][^2][^3][^6][^4][^1]

## Design choices

The structure keeps authentication and coarse-grained identity in Keycloak, while effective tenant-scoped and case-scoped authorization stays in PostgreSQL, which is the safer pattern for AML workflows that need assignment, review separation, and auditable access decisions.  The RLS policies use request-scoped runtime variables so the app can set tenant and user context per request before queries run, which is a standard PostgreSQL approach for shared-schema multi-tenancy.[^2][^3][^4][^7][^5]

## Keycloak notes

The realm export skeleton follows Keycloak’s JSON import/export model, where realm configuration is stored as JSON and can be imported into a new environment.  I also separated the composite-role plan into a companion markdown file because composite roles often need environment-aware setup after import, especially when client UUIDs or post-import client lookups are involved.[^8][^6][^9][^1][^2]

## Next step

Use the archive to initialize your RBAC foundation, then wire your API middleware so every authenticated request sets `app.current_tenant` and `app.current_user_id` before touching tenant-owned tables.  After that, the next useful step is to add a migration tool such as Alembic or Flyway and generate a `docker-compose.yml` that boots PostgreSQL and Keycloak with this starter config.[^10][^3][^4][^1]
