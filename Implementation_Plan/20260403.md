### Exploring Design Approaches

Given the constraints, the main architectural decision is **where and how the complex pattern matching (the core engine) runs**. Circular flow detection across tens of millions of nodes can be computationally brutal.

Here are 2 viable architectural approaches for the engine.

#### Option A: Database-Native Cypher Rule Engine (Recommended)
**How it works:**
The T+1 ETL orchestrator (e.g., Airflow, Dagster) pulls raw data, normalizes it, and checks it against an OFAC/FATF lookup table (Relational schemas). It then loads the nodes/edges into the Apache AGE graph. The Rule Engine is simply a module that issues parameterized `openCypher` queries (e.g., matching cycles `(a)-[*1..5]->(a)`, or temporal windows for rapid movement) directly against the database inside the batch window. Alerts are saved back to standard Postgres tables for the UI.
* **Pros:** Simplest operational architecture (everything lives inside Postgres). Highly extensible—adding a new AML typology just means writing a new Cypher template. Low infrastructure overhead.
* **Cons:** Native Cypher traversal algorithms for high-depth circular flows on 100M+ edges might strain the Postgres server's memory and compute. 
* **Why recommended:** It adheres ruthlessly to YAGNI (You Aren't Gonna Need It). We should push Postgres + AGE to its limit using the T+1 window before introducing external complexity.

#### Option B: Hybrid Pipeline with External Compute (The Big Big Data approach)
**How it works:**
The T+1 raw data hits an external distributed compute layer (e.g., Apache Spark GraphX or dedicated Python processing workers utilizing libraries like NetworkX). This layer runs the heavy algorithmic math (cycle detection, temporal groupings) in memory. The external compute layer generates the alerts, and then pushes the raw graph data + the alerts into PostgreSQL/Apache AGE purely so the Analyst UI can visualize and query them.
* **Pros:** Highly scalable. Easily handles deep recursion and complex time-window aggregations that relational/Cypher queries struggle with.
* **Cons:** Extremely high complexity. Introduces distributed compute infrastructure to maintain. Harder to debug than standard Cypher queries.

* **Missing Entity Data**: If a transaction arrives in the T+1 feed but the underlying entity logic is corrupted, the ETL will intercept it.

As the **Skeptic**, I am assuming this "production-grade" refactor will fail under real-world load. Here are the core weaknesses and failure modes identified in the proposed modular design:

By moving **Audit Logging** and **RBAC** to centralized middleware in a modular FastAPI app, we risk substantial overhead for high-frequency graph expansions.
*   **Failure Mode**: Every time an analyst clicks to "Expand Node" (traversing 1-2 levels), the backend performs a JWT decode, a DB lookup for permissions, and a synchronous write to an audit log. 
*   **Risk**: Under pressure (e.g., a major sanctioned entity investigation), UI latency will spike, leading to "Explorer Freezes."

##### 2. Apache AGE Connection Exhaustion
The refactor proposes a "Service Layer" to isolate AGE/Cypher logic.
*   **Failure Mode**: Unlike standard relational queries, Cypher traversals in Apache AGE are compute-heavy and hold connections longer. A modular API with `Depends(get_db)` on every route can easily exhaust the PostgreSQL connection pool if 10+ analysts run concurrent "Circular Flow" visual expansions.
*   **Risk**: A single complex "Super-Node" expansion could lock the entire backend for all users.

##### 3. Next.js "Hydration Mismatch" with Graph Libraries
Using Next.js App Router with **Cytoscape.js** (which is strictly DOM-dependent).
*   **Failure Mode**: Attempting to pre-render graph components on the server for "SEO/Performance" will cause hydration errors or forced client-side-only rendering which negates SSR benefits.
*   **Risk**: The "Explorer" page will have a massive LCP (Largest Contentful Paint) while the browser waits for the massive Cytoscape bundle and the `1..5` hop JSON response.

##### 4. Rule Engine "Memory Leak"
Moving Cypher templates to a `rules/` directory (JSON/YAML).
*   **Failure Mode**: If the service layer doesn't cache these templates or uses `eval()`-style string interpolation for parameters (id, threshold), we risk **Cypher Injection** and memory bloat from re-parsing complex templates on every trigger.
